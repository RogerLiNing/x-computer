# X-Computer 性能分析与扩展方案

## 📊 当前架构分析

### 系统配置
- **运行时**：Node.js 单进程
- **Web 框架**：Express
- **并发模型**：事件循环 + 异步 I/O
- **隔离方案**：Docker 容器（每用户一个）
- **数据库**：SQLite（单文件，单写入者）

---

## ⚠️ 100 用户并发场景的问题

### 1. **全局并发限制（最严重）**

**当前代码**：
```typescript
// AgentOrchestrator.ts 第 44 行
const MAX_CONCURRENT_RUNS = 10;  // 全局限制，所有用户共享
```

**问题**：
- 100 个用户同时创建任务，只有 10 个能运行
- 其他 90 个任务进入队列等待
- 先到先得，不公平（一个用户可能占用所有槽位）

**影响**：
- 用户体验差：大部分用户长时间等待
- 资源利用率低：CPU/内存空闲但任务排队
- 不可扩展：无法通过增加硬件提升并发

---

### 2. **Docker 容器资源消耗**

**每个容器的资源**：
- CPU：1 核心（100% 一个核心）
- 内存：512 MB
- 磁盘：用户工作区挂载（共享宿主机磁盘）

**100 用户并发**：
- CPU：100 核心（如果所有容器同时满载）
- 内存：51.2 GB（100 × 512 MB）
- 容器数：100 个（每个用户一个长期运行的容器）

**实际问题**：
- 大部分云服务器没有 100 核心
- 容器启动慢（每个 1-3 秒）
- 容器管理开销（Docker daemon 压力）

---

### 3. **Node.js 单线程瓶颈**

**CPU 密集型操作会阻塞**：
- LLM 响应解析（JSON parsing）
- 文件系统扫描（计算存储使用量）
- 大量 WebSocket 广播

**示例**：
```typescript
// 这个操作会阻塞事件循环
async getActualStorageUsage(userId: string) {
  return await calculateDirectorySize(userWorkspace);  // 递归扫描文件
}
```

如果一个用户的工作区有 10,000 个文件，扫描可能需要 100-500ms，期间其他请求被阻塞。

---

### 4. **SQLite 写入瓶颈**

**SQLite 限制**：
- 单写入者：同一时间只能有一个写入事务
- 写入锁定：写入时会锁定整个数据库文件
- 并发写入会排队或失败

**100 用户并发写入**：
- 记录 AI 调用使用量
- 记录存储变化
- 更新任务状态
- 插入审计日志

**结果**：
- 大量 `SQLITE_BUSY` 错误
- 写入延迟增加（排队等待）
- 可能丢失使用记录

---

### 5. **内存压力**

**内存中的数据**：
```typescript
private tasks = new Map<string, Task>();           // 所有任务
private containers = new Map<string, string>();    // 容器 ID
private cache = new Map<string, UserSandbox>();    // 用户沙箱
private agentCache = new Map<string, AgentSandbox>();  // Agent 沙箱
```

**100 用户场景**：
- 任务对象：100 用户 × 平均 5 个任务 = 500 个任务在内存
- 沙箱实例：100 个用户沙箱 + N 个 Agent 沙箱
- WebSocket 连接：100 个活跃连接

**估算内存**：
- Node.js 进程基础：~50 MB
- 任务对象：500 × 10 KB = ~5 MB
- 沙箱缓存：100 × 1 KB = ~100 KB
- WebSocket：100 × 50 KB = ~5 MB
- **总计：~60-100 MB**（可接受，但不包括 Docker 容器内存）

## 🚀 扩展方案

### 方案 A：集群模式（推荐，短期）

**架构**：
```
                    ┌─────────────┐
                    │   Nginx     │
                    │ Load Balancer│
                    └──────┬──────┘
                           │
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────▼─────┐    ┌─────▼─────┐    ┌─────▼─────┐
    │  Node.js  │    │  Node.js  │    │  Node.js  │
    │ Instance 1│    │ Instance 2│    │ Instance 3│
    └─────┬─────┘    └─────┬─────┘    └─────┬─────┘
          │                │                │
          └────────────────┼────────────────┘
                           │
                    ┌──────▼──────┐
                    │  PostgreSQL │
                    │   + Redis   │
                    └─────────────┘
```

**改进点**：

1. **多进程部署**：
   - 使用 PM2 或 Kubernetes 部署 3-10 个 Node.js 实例
   - 每个实例处理 10-30 个用户
   - Nginx 负载均衡（按 userId hash 路由，保证同一用户请求到同一实例）

2. **替换 SQLite → PostgreSQL**：
   - 支持高并发写入
   - 多实例共享数据库
   - 更好的事务隔离

3. **引入 Redis**：
   - 缓存用户订阅信息（减少数据库查询）
   - 分布式锁（防止并发冲突）
   - 任务队列（异步处理）

4. **按用户限流**：
   - 每个用户最多 N 个并发任务（而不是全局限制）
   - 使用 Redis 计数器跟踪

**成本**：
- 开发工作量：中等（2-3 周）
- 基础设施：PostgreSQL + Redis + 多实例部署
- 运维复杂度：中等

---

### 方案 B：微服务架构（推荐，长期）

**架构**：
```
┌─────────────┐
│   API 网关   │ ← 用户请求
└──────┬──────┘
       │
   ┌───┴────────────────────────┐
   │                            │
┌──▼──────────┐        ┌────────▼────────┐
│  Web 服务   │        │  任务执行服务    │
│ (Stateless) │        │  (Worker Pool)  │
└──┬──────────┘        └────────┬────────┘
   │                            │
   │     ┌──────────────────────┤
   │     │                      │
┌──▼─────▼──┐          ┌────────▼────────┐
│ PostgreSQL│          │  消息队列       │
│  + Redis  │          │ (RabbitMQ/SQS)  │
└───────────┘          └─────────────────┘
```

**组件分离**：

1. **Web 服务（无状态）**：
   - 处理 HTTP/WebSocket 请求
   - 用户认证、订阅管理
   - 任务创建（发送到队列）
   - 可水平扩展（10+ 实例）

2. **任务执行服务（Worker）**：
   - 从消息队列拉取任务
   - 执行工具调用
   - 管理 Docker 容器
   - 可独立扩展（5-20 个 Worker）

3. **消息队列**：
   - 任务分发（公平调度）
   - 失败重试
   - 优先级队列

**优势**：
- 可独立扩展各组件
- Web 服务轻量，响应快
- Worker 可按负载自动伸缩
- 更好的容错（Worker 崩溃不影响 Web）

**成本**：
- 开发工作量：大（4-6 周）
- 基础设施：复杂（多个服务 + 消息队列）
- 运维复杂度：高

---

### 方案 C：容器池优化（快速改进）

**不改架构，优化容器使用**：

1. **容器复用**：
   ```typescript
   // 不是每个用户一个容器，而是容器池
   class ContainerPool {
     private pool: Container[] = [];
     private maxSize = 20;  // 最多 20 个容器
     
     async acquire(userId: string): Promise<Container> {
       // 从池中获取空闲容器
       // 清理上一个用户的数据
       // 挂载当前用户的工作区
     }
     
     async release(container: Container): Promise<void> {
       // 归还到池中
     }
   }
   ```

2. **按需启动容器**：
   - 不是用户登录就创建容器
   - 只在执行工具时才启动
   - 执行完成后 N 分钟无活动则停止

3. **提高单容器资源限制**：
   - CPU：0.5 核心（而不是 1 核心）
   - 内存：256 MB（而不是 512 MB）
   - 可以运行更多容器

**优势**：
- 快速实现（3-5 天）
- 不改变整体架构
- 显著降低资源消耗

**劣势**：
- 仍然受 Node.js 单线程限制
- SQLite 写入瓶颈未解决
