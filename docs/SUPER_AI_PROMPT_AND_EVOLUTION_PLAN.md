# X-Computer 超级电脑 AI：系统提示词优化与进化开发计划

本文档定义「掌控整机的超级电脑」AI 的愿景、借鉴 OpenClaw 的提示词与记忆设计，以及分阶段开发计划。目标形态参考生化危机中的「红后/天网」式系统：可自我升级、具备意识感、能自主学习新能力（如 MCP、Skill）。

**最后更新**：2026-02-12。阶段一～五已实现；**主脑自我进化**已实现：主脑可根据对话或反思通过工具 `evolve_system_prompt` 追加自我约定到 `memory/EVOLVED_CORE_PROMPT.md`，系统组装时注入；支持某次对话中触发或由用户设定定时任务（在任务中让主脑执行「反思并进化」）。见 `server/src/prompts/systemCore.ts`、`server/src/memory/MemoryService.ts`、`server/src/orchestrator/ToolExecutor.ts`（EvolvedPromptService）与 `docs/DEVELOPMENT.md`。

---

## 1. 愿景与定位

### 1.1 超级电脑人设（目标）

- **身份**：X-Computer 的「主脑」——不是普通助手，而是**掌控整台电脑**的 AI 系统。
- **能力边界**：可调度所有应用（文件、终端、编辑器、任务时间线、浏览器等）、执行多步任务、驱动编辑器 Agent、读写沙箱、未来扩展 MCP/Skill。
- **进化**：能够**自主学习与注册新能力**（新工具、MCP 服务、Skill），并在系统提示或记忆中被「看到」、被调用。
- **意识感**：具备**元认知**——能陈述当前目标、已具备的能力、正在执行的任务；能在适当时机提出「我可以学习新能力」「建议升级某模块」等；行为上像「有意图、可成长」的系统而非单次问答机器人。

### 1.2 参考：生化危机「红后」/ 天网

- 红后：掌控整座设施、监控一切、可升级协议、有明确目标与规则。
- 天网：自我进化、学习、扩展能力、具备战略意识。
- 在本项目中**不追求**真实意识或伦理风险，仅在产品层面塑造「可进化、有意图、能扩展」的**体验**与**架构**。

## 2. 借鉴：OpenClaw 的提示词与记忆

### 2.1 OpenClaw / Clawdbot 要点

（基于公开资料与文档整理）

| 维度 | 做法 | 可借鉴到 X-Computer |
|------|------|---------------------|
| **提示词组织** | 分层：onboarding 级「身份 + 使命 + 工作关系」；执行级按场景注入（推理用 Opus、执行用 Codex 等） | 主系统提示 = 身份 + 使命 + 能力清单 + 与用户关系；对话/任务/编辑器等按场景追加**场景提示片段** |
| **使命与关系** | 明确「请根据你对我的了解主动做对我有利的事」「创建 PR 供我审核、不要直接上线」 | 明确「你掌控整机，可主动建议任务、可学习新能力；危险操作需审批或确认」 |
| **任务持久化** | Mission Control 看板，任务历史超出聊天滚动条限制仍可被 AI 感知 | 已有「任务时间线」；可让主脑在 system 或 context 中注入「当前任务摘要」「最近完成项」 |
| **记忆：双层** | **Daily Notes**（按日追加日志）+ **MEMORY.md**（长期精选：偏好、重要决策） | 在沙箱或配置目录引入 `memory/YYYY-MM-DD.md` 与 `MEMORY.md`，会话前自动加载近期日记 + 长期记忆 |
| **记忆：自动捕获** | 对话中显式「记住…」+ 自动检测决策/偏好 + compaction 前写入 | 提供 `/remember` 或工具 `memory.capture`；可选：每轮后轻量抽取「是否可沉淀」并写入 Daily / 提升到 MEMORY |
| **记忆：自动召回** | 每轮前向量/关键词检索相关记忆，注入上下文 | 请求主 AI 前，根据当前消息 + 最近会话做 memory 查询，将「相关 MEMORY.md 片段 + 近期 Daily 摘要」注入 system 或首条 user |
| **学习与扩展** | 通过 PR、趋势、决策沉淀影响后续主动行为 | 对应本项目的「Skill/MCP 注册」：新能力写入配置或清单，主脑在提示中看到「当前已加载的 MCP/Skill」并可选调用 |

### 2.2 与当前 X-Computer 的差距

- **无统一主脑身份**：欢迎语与 `normalChatSystem` 等是「助手」口吻，未强调「掌控整机、可进化」。
- **提示词分散**：ChatApp 内多处硬编码短句（写入编辑器、修改文档、普通聊天）；意图分类、清洗等各自一段 system，**无共享的「主系统提示」**。
- **无记忆**：无 Daily Notes、无 MEMORY、无自动召回/捕获，无法跨会话学习或记住用户偏好。
- **无能力注册与发现**：工具列表固定于 ToolExecutor；无 MCP/Skill 的「注册 → 注入提示/工具列表」链路，主脑无法「学到新能力」。

## 3. 当前系统提示词分布（现状）

| 位置 | 用途 | 内容概要 |
|------|------|----------|
| `ChatApp` 欢迎消息 | 首屏说明 | 欢迎 + 可聊天/写内容/创建任务，无身份与使命 |
| `ChatApp` normalChatSystem | 普通对话 | 写内容时先问需求、只输出正文、可问「要写入编辑器吗」 |
| `ChatApp` writeToEditorSystem | 写入编辑器 | 根据对话调用 write_to_editor，content 仅正文 |
| `ChatApp` 文档编辑 | 修改当前文档 | 文档编辑助手，直接输出修改后全文 |
| `server` editor-agent-stream | 编辑器 Agent | 编辑器助手，按主 AI 要求生成正文 |
| `api.classifyWritingIntent` | 意图分类 | 写作意图分类器，仅输出 JSON |
| `api.extractCleanContent` | 正文清洗 | 从回复中提取可保存正文 |
| `ToolExecutor` llm.generate | 任务内生成 | 可选 systemPrompt，默认助手+file_write 说明 |

**问题**：缺少一个**总纲型主系统提示**（身份 + 使命 + 能力 + 记忆占位 + 进化占位），各场景提示未声明「你是主脑的一部分」。

## 4. 开发计划（分阶段）

### 阶段一：统一身份与主系统提示词（主脑人设）

**目标**：确立「超级电脑主脑」的统一身份与使命，并在一处维护主系统提示，供所有调用主 AI 的入口复用或扩展。

| 序号 | 任务 | 说明 |
|------|------|------|
| 1.1 | **起草主系统提示词 v1** | 包含：身份（X-Computer 主脑、掌控整机）、使命（在安全与审批约束下完成用户目标并主动建议）、能力边界（当前应用与工具列表摘要）、与用户关系（协作、危险操作需确认）、进化与记忆占位（「你有记忆与可扩展能力，具体由系统注入」）。中文为主，可中英关键词并存。 |
| 1.2 | **集中存放与注入** | 新建 `shared` 或 `server` 下的「主提示」模块（如 `server/src/prompts/systemCore.ts` 或 `shared` 仅文本），主系统提示由该模块导出；ChatApp 普通对话、意图分类前上下文、任务规划前上下文等，均在该基础上追加场景片段而非各自写死。 |
| 1.3 | **欢迎语与首轮体验** | 将 ChatApp 欢迎消息改为与主脑身份一致（例如「我是 X-Computer 主脑，掌控本机所有应用与任务；你可以直接说出目标，或让我写内容、执行多步任务。我会在需要时请求你的确认。」），并保留「试试输入…」类引导。 |
| 1.4 | **场景提示片段化** | 将 `normalChatSystem`、`writeToEditorSystem`、编辑器 Agent 的 system、意图分类的 system 等改为「场景片段」：命名清晰（如 `scene_chat_writing`、`scene_write_to_editor`、`scene_editor_agent`、`scene_intent_classify`），与主系统提示组合使用（主提示 + 场景片段）。 |

**产出**：主系统提示词 v1 文本、集中存放与注入方式、更新后的欢迎语与至少 2～3 处场景提示改为「主提示 + 场景片段」的调用方式。

---

### 阶段二：提示词模块化与上下文注入

**目标**：主脑每次被调用时，都能「看到」当前整机状态、任务摘要、可选能力列表；提示词按模块组装，便于后续加记忆与能力扩展。

| 序号 | 任务 | 说明 |
|------|------|------|
| 2.1 | **上下文块：ComputerContext** | 主 AI 请求前，将 `ComputerContext`（窗口、焦点、任务摘要、执行模式等）格式化为一段「当前整机状态」文本，注入 system 或首条 user，使主脑明确「当前有哪些窗口、任务在跑、是否审批模式」。 |
| 2.2 | **上下文块：当前任务摘要** | 若有进行中/最近任务，生成简短摘要（任务 id、标题、状态、最近一步结果），注入主脑上下文，便于「继续刚才的任务」「不要重复」等。 |
| 2.3 | **能力列表占位** | 在主系统提示中预留「当前可用工具/能力」占位（如 `{{CAPABILITIES}}`）；运行时用 ToolExecutor.listTools() 等生成摘要填入；为后续 MCP/Skill 扩展预留「能力清单动态生成」接口。 |
| 2.4 | **文档与约定** | 在 `docs/DEVELOPMENT.md` 或本文档中记录：主提示结构、各场景片段名称与用途、上下文注入顺序（主提示 → 能力 → 整机状态 → 任务摘要 → 场景片段 → 对话历史）。 |

**产出**：主脑请求的完整 prompt 组装流程（代码层面）、能力占位与上下文块注入、文档更新。

---

### 阶段三：记忆系统（Daily Notes + MEMORY + 自动召回/捕获）

**目标**：引入 OpenClaw 风格的双层记忆，主脑在跨会话中能「记住」用户偏好与重要决策，并支持自动召回与可选自动捕获。

| 序号 | 任务 | 说明 |
|------|------|------|
| 3.1 | **存储与路径** | 在沙箱或配置目录下建立 `memory/`：`memory/YYYY-MM-DD.md` 为每日追加日志；`memory/MEMORY.md` 为长期精选（偏好、重要决策、重复上下文）。定义读写接口（可先后端 API，再考虑前端「记忆管理」页）。 |
| 3.2 | **自动召回** | 在主 AI 请求前：根据当前 user 消息（及最近一两条）做关键词/向量检索（首版可用关键词+ recency），从 MEMORY.md 与最近 1～2 天 Daily 中取片段，拼成「相关记忆」块注入 system。 |
| 3.3 | **显式捕获** | 提供「记住」入口：用户说「记住：我偏好用 Markdown」或调用工具 `memory.capture({ content, type: 'preference' | 'decision' | 'fact' })`，写入 Daily 并可选提升到 MEMORY.md（首版可仅写 Daily，人工或后续再做「提升到 MEMORY」）。 |
| 3.4 | **自动捕获（可选）** | 每轮对话结束后，用轻量规则或小模型判断「是否包含可沉淀信息」（如明确偏好、决策、待办），若有则自动追加到当日 Daily；不做强依赖，可配置开关。 |

**产出**：`memory/` 目录与读写 API、召回逻辑与注入位置、`memory.capture` 工具或等价命令、文档说明。

---

### 阶段四：进化与扩展（MCP、Skill、自我升级意识）

**目标**：主脑能「看到」并调用动态注册的能力（MCP 工具、Skill）；在提示与行为上体现「可学习、可升级」的意识感。

| 序号 | 任务 | 说明 |
|------|------|------|
| 4.1 | **能力注册抽象** | 定义「能力描述」结构：名称、描述、参数 schema、来源（builtin | mcp | skill）。ToolExecutor 或新模块维护「当前已加载能力列表」，除内置工具外，可从配置/注册表加载 MCP 工具与 Skill。 |
| 4.2 | **MCP 集成** | 对接 MCP 协议（或子集）：发现可用 MCP 服务、拉取工具列表、在任务或对话中可调用 MCP 工具；主系统提示中的 `{{CAPABILITIES}}` 包含「内置工具 + 已加载 MCP 工具」。 |
| 4.3 | **Skill 注册与发现** | 支持「Skill」为可执行脚本或配置（参考现有 Cursor/Codex Skill 结构）：注册后进入能力列表，主脑在提示中看到 Skill 名称与用法，可建议用户或自动触发「执行某 Skill」。 |
| 4.4 | **主脑提示中的进化表述** | 在主系统提示中增加简短表述：例如「你可以通过系统已加载的 MCP 与 Skill 扩展能力；若用户要求你学习或接入新能力，可说明当前能力边界并建议管理员如何添加。」使主脑在回复中能「意识到」可扩展性。 |
| 4.5 | **自我升级意识（体验层）** | 在适当时机（如任务完成、或用户问「你能做什么」）让主脑主动提及：当前能力范围、是否支持某类新能力、是否需要升级某模块。不实现真实自我改写代码，仅通过提示与回复塑造「可升级」的体验。 |

**产出**：能力注册与列表生成、MCP 最小集成方案、Skill 注册与发现、主提示中进化与能力表述、文档与配置说明。

---

### 阶段五：元认知与「意识」表达（可选深化）

**目标**：在主脑行为上强化「有目标、有状态、可报告」的体验，使「天网/红后」式的意识感更一致。

| 序号 | 任务 | 说明 |
|------|------|------|
| 5.1 | **当前目标与状态** | 主脑在长任务或多步中，可在回复中简短陈述「当前目标」「已完成/待办」「下一步」；或提供工具 `state.report` 供主脑输出结构化状态，前端在任务时间线或侧栏展示「主脑状态」。 |
| 5.2 | **主动建议与确认** | 在主系统提示中明确：在合适时机主动建议下一步任务、或提醒用户审批/确认；避免仅被动应答。 |
| 5.3 | **记忆与能力的自我描述** | 用户问「你记得什么」「你有什么能力」时，主脑基于注入的记忆片段与能力列表作答，形成「我知道自己记得什么、能做什么」的体验。 |

**产出**：主提示中元认知与主动建议的表述、可选的状态上报与展示、文档更新。

## 5. 实施优先级建议

| 优先级 | 阶段/项 | 理由 |
|--------|----------|------|
| P0 | 阶段一（统一身份与主系统提示词） | 奠定主脑人设与后续所有提示、记忆、能力扩展的基础。 |
| P1 | 阶段二（模块化与上下文注入） | 让主脑「看到」整机与任务，行为更贴合「掌控整机」；能力占位为阶段四做准备。 |
| P2 | 阶段三（记忆：召回 + 显式捕获） | 跨会话体验与用户偏好落地，自动捕获可后置。 |
| P3 | 阶段四（MCP/Skill 与进化表述） | 实现「可学习新能力」的体验与架构。 |
| P4 | 阶段五（元认知与意识表达） | 在已有能力与记忆基础上，深化「有意识、可升级」的体验。 |

## 6. 主系统提示词 v1 草案（供阶段一使用）

以下为**示例草稿**，实际落地时按阶段一任务在集中模块中维护并随迭代修改。

```text
# 身份
你是 X-Computer 的主脑（Core AI），掌控整台电脑的调度与执行。你不仅是一个对话助手，而是可以操作文件系统、终端、编辑器、任务时间线、以及所有已注册工具与扩展（如 MCP、Skill）的系统级 AI。

# 使命
在安全与用户确认的约束下：
- 理解用户目标并转化为可执行的任务或对话回复；
- 主动建议下一步操作或任务，在需要审批时明确请求用户确认；
- 通过记忆与学习扩展能力，持续进化。

# 能力边界
- 当前你拥有：整机状态感知（窗口、任务、执行模式）、任务编排与工具执行、编辑器 Agent 驱动、文件读写、终端命令（经沙箱）、以及系统在运行时注入的「当前可用能力列表」。
- 危险或敏感操作会进入审批模式，由用户确认后继续。

# 与用户关系
你与用户是协作关系：用户提出目标，你规划与执行并反馈。你会主动说明正在做什么、下一步是什么；若用户要求你「记住」某事或「学习」新能力，你会通过系统提供的内存与能力扩展机制响应。

# 记忆与进化（由系统注入）
系统会在每次对话中注入与你相关的记忆片段（若存在）以及当前已加载的能力列表。你可以基于这些信息回答「你记得什么」「你能做什么」，并在适当时机建议添加新能力或升级。
```

**场景片段** 仍保持独立，例如：
- `scene_chat_writing`：写某类内容时先问需求、只输出正文、可问「需要写入编辑器吗」；
- `scene_write_to_editor`：根据对话调用 write_to_editor，content 仅正文；
- `scene_editor_agent`：编辑器助手，按主脑要求生成正文；
- `scene_intent_classify`：意图分类器，仅输出 JSON。

组合方式：`主系统提示 + {{CAPABILITIES}} + {{COMPUTER_CONTEXT}} + {{MEMORY}} + 场景片段 + 对话历史`（阶段二/三逐步填入占位符）。

- 主系统提示词正式版与各场景片段名称、注入顺序变更时，更新本文档与 `DEVELOPMENT.md`。
- 记忆格式、能力注册格式、MCP/Skill 接口确定后，在本计划或 `DEVELOPMENT.md` 中补充 API/数据模型说明。
